## Learned User Preferences

- Always communicate with the user in German (UI copy, explanations, and coach-facing text).
- **Git (Dev):** Push changes directly to `master` — no feature branches or separate PRs unless the user asks otherwise.
- Onboarding: Ziel single-select; Hindernis multi-select; order branche → … → rolle (mid-flow) → … → vorname + firmenname (steps 9–10, before auth); „Rundherum" questions (Verwaltung, Buchhaltung, Wissen, etc.) for all team sizes including Solo.
- Phase 1 coach: four Etappen — (1) active pain point: only matching `idea_card`, one integrated Ablauf (never split Text/Bilder/Layout as options); (2) „noch etwas?" before catalog; (3) all remaining idea_cards + click-to-explain + final „noch eine interessant?"; (4) Reihenfolge + `prepare_phase(analyse)` + `phase_complete` only when user ready; no „typischer Tag"; `<canvas_update>` mandatory when claiming canvas; live UX: chat text first, loader on opening tag, then apply + „fertig"; intro → „Lass uns gleich starten:" → first diagnostic question in message 2; deep questions one at a time, but short clarifications can be bundled as multi-step `<options>` questions rendered one at a time.
- Chat coach: plain text on white; no `---`, dialog prefixes, or meta filler; no markdown tables (numbered lists for comparisons); client keeps raw `<options>` in message state, DB/store gets stripped text; no extra visible Canvas/status assistant bubble; Claude/Anthropic must stream live, keep pre-tool-call text visible, then continue after the tool without `<stream_reset>`; image attachments = thumbnail preview only (no filename/OCR badge); attachment metadata hidden via `<user_attachments>` — no OCR or filenames in visible bubbles; Strg+V image paste in coach + workflow input (no placeholder hint); voice mic beside send (Voxtral), recording status must not shift input layout; help/support bubble `fixed bottom-8 right-8` at screen viewport (not inside chat panel or header).
- Chat/canvas responsive: `CanvasInfoPanel` (phase pill + project name) only at `xl+`; below `xl` use sidebar `ProjectMenu`; chat panel default narrow (360px) below `xl`, wide (600px) at `xl+`; history dropdown shows all project chats (scrollable), closes on outside click and when opening hamburger menu; roadmap canvas fills area right of chat via absolute inset (`left: 0` when chat closed); chat shell/panel `100dvh` with `bottom-8`; ChatInput/OptionsCard pinned footer outside message scroll (`shrink-0`); interactive elements app-wide need `cursor: pointer` (Tailwind v4 base layer in `globals.css`).
- Landing page (marketing): customer-facing German; light `bg-slate-50` + `bg-grid` (no dark hero gradients); no scroll phase/roadmap connector line; story Verstehen → Planen → Bauen → Live-Schalten; avoid deployen, Canvas, Workflow-Engine, n8n, Human-in-the-loop on LP — use Plan/Roadmap/Umsetzen/Live-Schalten; vertical pages e.g. `/immobilienmakler` (`MaklerLanding`).
- Phase transition: manual button after summarize; phases 2+ via hidden init from `lib/phase-welcome.ts` / `lib/session-kickoff.ts` — never Phase 1 onboarding intro on phase switch or empty session.
- Phase 2 (Analyse & Plan): recap + start-point via `<options>`; Klärfragen (Ist-Ablauf, Tools, einfache Entscheidungen) as multi-step `<options>` one at a time; bei komplexen Lösungsbereichen (Exposé, mehrstufige Docs) konkreten Ablauf nummeriert im Fließtext — keine vagen Lösungswege-`<options>` (A/B/C wie „KI füllt Vorlage / sendet direkt / nur Vorlage" verboten); danach nur Ja/Nein-Gate; bei „Ja, passt" sofort `<workflow_plan>`; tools per pain, use_cases = status quo only; no credentials/deploy/build in analyse; coach writes canvas via `<canvas_update>` directly.
- Phase 3 (Umsetzung): no finished workflows at start — ask which plan to build first, then live workflow tool-call build; coach knows workflow editor UI; no re-interview, A/B/C, or tool re-discovery; deploy cards at Umsetzung node only; ~90% modal with margin; React Flow; fit-view on workflow open; custom minimap styled like canvas nodes (not blue boxes); trailing `+` connector after terminal nodes (n8n-style); nodes icon-only with label below, drag follows cursor; n8n catalog SSOT (categorized picker, auto-pick on create, MCP `search_nodes` supplements search); real tool icons; red credential borders; Google/Microsoft OAuth = inline „Mit … verbinden" (no credential dropdown/popup/API-key for OAuth2 types); Deploy in modal only; rich per-node config; per-node German purpose note (not generic EN node description); AI Agent/summarization chain sub-node ports (model/memory/tools); bottom-center editor chat as Setup-Coach — same coach context as main chat, output stays in editor only; AI edits preserve edges + full workflow overview; large popups for node switch/add; first node must be trigger.
- Never refactor `RoadmapCanvas` curved path, phase checkpoints, or side PhaseNode TitleBox — only content rails, card placement, and doc-first dedup; canvas/memory blobs only facts from the conversation — no speculation or invented fields; terminal logs use `[agent-sync][canvas|memory][event]`; auth/OAuth redirects must keep the user on the same origin they started from (prod, preview, or localhost); restart dev server after `.env.local` changes.
- **AI cost & pricing:** Haiku 4.5 for diagnose coach, Sonnet 5 for analyse/umsetzung + `lib/strategy.ts`; avoid Opus for chat; planned credits unify coach+build — deployed workflows are the retention lever (low ongoing credit use).
